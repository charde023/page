결론 (TL;DR)

AI는 이미 똑똑하다. 문제는 사람이 못 다루는 것. 똑똑한 경주마를 내 뜻대로 달리게 미리 환경을 세팅하는 게 하네스 엔지니어링이고, 그 정점이 서로 소통하는 멀티 에이전트 팀이다.

한 줄로 보는 4단계

단계 한 줄 정의 핵심 도구
프롬프트 엔지니어링 AI에게 말하는 법 롤 지정, 예시, 역질문
컨텍스트 엔지니어링 AI가 볼 수 있는 모든 정보를 마련 개인 맞춤 지침, 메모리, 파일
하네스 엔지니어링 AI가 알아서 일하게 환경을 세팅 CLAUDE.md, 스킬, 서브 에이전트
멀티 에이전트 (팀) 여러 에이전트를 고용·협업 서브 에이전트, 멀티 에이전트 팀

오늘 챙겨야 할 5가지

# 메시지 한 줄
1 하네스는 새로운 게 아니다 목표는 단 둘 — 프롬프트 재사용 + 정보 재사용
2 CLAUDE.md = 작업 폴더 공지사항 폴더마다 두면 AI가 채팅 열 때마다 먼저 읽음
3 에이전트는 다 같은 클로드코드 능력치는 같고 지침만 다름. 메인이 서브를 고용
4 멀티 에이전트 팀 = 서로 소통 팀원이 회의·피드백 → 토큰·시간 낭비 없이 피봇
5 리뷰어 + 중간 산출물 운영의 핵심. 피드백 루프와 단계별 산출물을 꼭 정의

단일 vs 멀티 vs 팀

서브 에이전트 멀티 에이전트 멀티 에이전트 팀
구조 메인이 1명 고용 메인이 여러 명 따로 고용 한 세트(팀)로 고용
실행 단독/병렬 병렬 동시, 각자 결과 제출 병렬 + 팀원끼리 소통
강점 역할 전담 시간 단축 회의·상호 피드백으로 품질↑

강의는 어떤 내용이었나

강사 허세임(본명 허승현, AI 강의·AX 컨설팅 3년차)이 "AI와 협업하는 방법"을 주제로 진행한 2시간짜리 라이브. 클로드코드(Claude Code)를 기준으로 프롬프트 → 컨텍스트 → 하네스 엔지니어링의 계단을 한 칸씩 밟아 올라가, 최종 목적지인 멀티 에이전트 팀 활용에 도달한다.

핵심 메시지는 일관적이다 — "하네스 엔지니어링은 거창한 신기술이 아니라, 프롬프트와 정보를 재사용하도록 미리 세팅하는 것." 여행 계획 세우기와 막장 쇼츠 드라마 제작이라는 두 친숙한 예시로 추상 개념을 끝까지 손에 잡히게 풀어낸다.

교시 구간 내용
1교시 기초 프롬프트 → 컨텍스트 → 하네스 엔지니어링, 클로드 vs 클로드코드, CLAUDE.md
2교시 에이전트 활용 서브 에이전트 → 멀티 에이전트 → 멀티 에이전트 팀, 운영 주의사항 3가지

0. 한눈에 보는 하네스 엔지니어링

  1. 프롬프트 엔지니어링 — 롤 지정·예시·출력형식, 그리고 역질문("네가 나한테 질문해라")
  2. 컨텍스트 엔지니어링 — 나에 대한 정보를 미리 넣어 "나를 잘 아는 사람"처럼 답하게
  3. 컨텍스트 오염 관리 — 대화는 3턴 내로, 채팅창은 자주 새로, 불필요한 정보는 빼기
  4. CLAUDE.md — 작업 폴더별 공지사항. AI가 채팅 열 때마다 자동으로 읽음
  5. 서브 에이전트 — 미리 지침을 적어둔 에이전트를 메인이 고용 (하청의 하청)
  6. 멀티 에이전트 팀 — 여러 명을 한 팀으로 고용, 서로 소통하며 협업
  7. 운영 원칙 — 역할 설계 + 리뷰어 + 중간 산출물 정의

1. 프롬프트 → 컨텍스트 → 하네스

1-1. 프롬프트 엔지니어링 (말하는 법)

"가비지 인, 가비지 아웃." 좋은 질문이 좋은 답을 만든다. 익숙한 기법들 — 롤 지정("너는 30년 차 실리콘밸리 개발자야"), 예시 제공, 출력 형식 지정. 강사가 가장 사랑하는 건 역질문: "내가 너한테 묻는 게 아니라, 네가 나한테 질문해라." 내 생각을 확장·구체화한다.

팁: 단, 지금 AI는 충분히 똑똑해서(사투리·오타도 이해) 프롬프트 엔지니어링의 비중은 예전보다 줄었다.

1-2. 컨텍스트 엔지니어링 (정보를 마련)

왜 프롬프트를 길게 쓰는가? 나에 대한 정보를 보내 내 입맛에 맞는 결과를 얻으려고. 챗GPT·클로드는 똑똑하지만 나를 모른다. 반면 (지능은 낮아도) 나를 잘 아는 엄마는 더 입맛에 맞는 여행지를 추천한다.

무릎 사례 — 프롬프트에 "무릎이 아프다"는 말이 없어도, 내 정보를 미리 넣어둔 클로드코드는 "무릎 통증을 고려해 많이 걷지 않는 여행지로 찾아볼게"로 시작한다.

컨텍스트 = 채팅창(컨텍스트 윈도우)에 들어가는 모든 것: 내가 친 프롬프트 + 미리 적어둔 시스템 프롬프트 + 그동안의 대화(메모리) + 참고 파일 + 웹 검색 결과.

1-3. 컨텍스트 오염 (다 넣으면 안 된다)

증상 원인
여행 질문에 개발자 느낌으로 답함 개인 지침을 과하게 인식
힐링 여행 물었는데 회사 출장지 추천 업무 정보가 섞임 = 컨텍스트 폴루션

관리법 2가지 — ① 티키타카 횟수 줄이기(3턴 넘으면 새 대화창) ② 불필요한 컨텍스트 빼기. 내 정보는 미리 넣되 오염은 막는, 적재적소 세팅이 곧 하네스 엔지니어링.

1-4. 하네스 엔지니어링 정의

하네스(harness) = 마구(馬具), 강아지 목줄. 똑똑하지만 길들여지지 않은 경주마(AI)를 내 뜻대로 달리게 미리 세팅하는 것.

중요 목표는 단 둘 — ① 프롬프트의 재사용 (매번 "무릎 아프다" 말하지 않기) ② 정보의 재사용 (나에 대한 정보를 미리 넣어 AI가 갖다 쓰게).

기초 요소: CLAUDE.md · 슬래시 커맨드 · 스킬 · MCP · 훅 · 서브 에이전트. (오늘은 CLAUDE.md와 에이전트 위주.)

2. 클로드 vs 클로드코드, 그리고 CLAUDE.md

2-1. 햄과 햄스터처럼 다르다

클로드 = AI 어시스턴트 클로드코드 = AI 에이전트
역할 질문하면 답변 ("입만 터는 애") 명령하면 직접 수행해 결과물 제공
예시 챗GPT, 제미나이, 클로드 웹 클로드코드, 코덱스, 안티그래비티

AI 에이전트란 — 목표를 주면 스스로 단계를 쪼개 계획하고, 도구를 고르고, 중간 결과를 보며 다음 행동을 정하고, 끝날 때까지 스스로 움직이는 시스템.

클로드 3가지 버전: 웹 / 데스크탑(초보 추천) / CLI(개발자 버전, 보통 "클로드코드"). 모델은 한도 절약 위해 Sonnet으로.

2-2. 작업 폴더 지정이 가장 중요

클로드와 클로드코드의 가장 큰 차이 = 작업(프로젝트) 폴더 지정. 프로젝트별로 폴더를 열어 작업한다.

2-3. CLAUDE.md = 폴더별 공지사항

CLAUDE.md 파일이 폴더에 있으면 AI가 무조건 먼저 읽고 시작한다. 업무 폴더엔 업무 설명, 여행 폴더엔 여행 정보를 따로 둬서 컨텍스트 오염을 막는다.

만드는 법 — 직접 쓰지 말고 시키기: "이 폴더에서 여행을 할 건데, 들어갈 내용을 나에게 질문해주고, 내가 답하면 네가 CLAUDE.md를 작성해줘."

  • 전역 CLAUDE.md(컴퓨터 전체) vs 폴더별 CLAUDE.md가 따로 있다.
  • 심화: LLM 위키 / 온톨로지 — 개인·업무 정보를 축적하고 연결해 AI가 참조하게.

3. 서브 에이전트

미리 할 일을 처음부터 끝까지 적어둔 에이전트. 나와 대화하는 애 = 메인 에이전트, 메인이 또 다른 클로드코드를 고용하면 서브 에이전트(외주의 외주). 공식 기능명일 뿐, 기능은 똑같은 클로드코드.

만드는 법: "여행 계획용 서브 에이전트를 만들어줘. 앞으로 여행 계획 세울 때마다 쓰게."

포인트 내용
능력치 메인 = 서브, 동일. 지침만 다름
역할 각자의 역할 지정 가능, 병렬 작업에 유용
이름 AI가 지어줌
소통 서브 실행 중에도 메인과 대화 가능 (메인이 서브에 전달)

저장 위치 — .claude/agents/에 생성: - 작업 폴더 안 → 그 폴더에서만 사용 - 중앙 폴더 → 모든 폴더에서 사용. (안 보이면 중앙에 저장된 것 → "이 프로젝트에 저장해")

4. 멀티 에이전트 & 멀티 에이전트 팀

4-1. 멀티 에이전트

메인이 여러 명을 고용, 병렬 동시 실행 후 각자 결과 제출. 설계 2가지 — 용도별 분담(조사/예산/동선) 또는 같은 작업 동시 처리(포르투갈·스페인·이탈리아 동시 조사 → 시간 단축).

장면: 사례 — AI 쇼츠 드라마: 4개월 전 멀티 에이전트로 만든 막장 드라마가 3화에서 37,000 조회수. 프로듀서·작가·캐릭터 디자이너·이미지 크리에이터 등에게 지침을 미리 적어두고 "드라마 한 편 하자"로 대본→이미지→영상→편집→업로드까지.

4-2. 멀티 에이전트 팀 — 서로 소통한다

한 명씩 따로가 아니라 한 팀으로 고용. 가장 큰 차이 = 팀원끼리 회의·소통.

상황 단순 멀티 멀티 에이전트 팀
시장 조사 중복 발견 각자 토큰 다 쓰고 끝나서야 "중복이네" 진행 중 "비상, 중복!" 회의 → 즉시 피봇
품질 개선 결과 따로 모음 VC 페르소나가 피드백 → 디벨롭 루프

발언: "많은 기법을 켜켜이 쌓는 것보다, 이것(멀티 에이전트 팀)만 할 줄 알아도 큰 프로젝트에서 많은 진전이 있다."

4-3. 세팅 — AI는 최신 기능을 모른다

  • 환경 변수 = 1 이어야 작동.
  • 주의: 클로드코드는 이 기능이 있다는 걸 모른다 (학습 시점까지만 앎). → "없다고 하지 말고 공식 문서를 조사해서 제대로 해라"라고 먼저 시키고, 이 지시를 CLAUDE.md에 명시.
  • 종료 시 메인에게 "팀을 다 종료해줘". 토큰 소모 큼(Max 요금제 권장).

플랫폼 제약 — 실시간 화면 분할은 터미널 등 일부만. 윈도우·데스크탑은 사이드바 "백그라운드 태스크"에 인원수로 확인. 윈도우는 git 설치 필요.

4-4. 팀 구성 2가지

  1. 미리 만들어 두기 — 다작/반복 업무 (드라마처럼 4개 에이전트 사전 구성)
  2. TF처럼 임시 구성 — 일회성. "여행 조사 에이전트 1명 + 나머지는 네가 임시로 꾸려서 팀으로 짜줘."

5. 운영 주의사항 3가지 — 오늘의 핵심

# 원칙 설명
① 목적에 따라 역할 설계 대량 처리(같은 작업 분할로 시간 단축) vs 다양한 작업 분담
② 리뷰어를 꼭 넣어라 중요 체크리스트로 피드백 주며 루프를 돌려 품질↑. 검증 페르소나도 가능
③ 중간 산출물을 명확히 정의 대본→영상→편집처럼 인풋→아웃풋 단계를 정의해야 AI가 안 놓침

목표: 목적은 "화면이 쪼개지네" 구경이 아니라 AI와 함께 일을 끝내는 것. 끝까지 퀄리티를 지키려면 ②리뷰어와 ③중간 산출물을 반드시 챙겨라.

6. 자주 나온 질문 (Q&A)

질문 답변
리뷰어, 꼭 필요한가? 무조건. 한 번 만든 에이전트·스킬은 늘 보강이 필요('깎는다'). 피드백 플로우로 못 미치면 다시/업그레이드
에이전트 vs 스킬 차이? 형식(MD/폴더)으로 나누지 말 것. 에이전트 = 사람의 역할(별도 창, 결과만 받음), 스킬 = 능력(메인 내 실행, 범용·교차 사용)
초보자 구성법? 먼저 티키타카로 업무 하나를 직접 완성 → "다음에도 쓰게 네가 구성해봐" → 하네스용으로 전환
대규모 엑셀 반복? 웬만하면 스크립트로 처리(토큰 절약). AI 판단이 꼭 필요한 것만 스킬·에이전트로
CLAUDE.md vs 스킬, 어디 공들여? 둘 다
전역 CLAUDE.md 수정? 중앙 폴더 안 열어도, 아무 폴더에서 "중앙 CLAUDE.md 업데이트하자"

증상 대응
AI가 자꾸 엉뚱한 폴더에서 작업 작업 폴더 지정부터. 클로드코드는 "연 폴더"를 기준으로 동작한다
매번 같은 맥락을 다시 설명 폴더에 CLAUDE.md(폴더별 공지)로 규칙·맥락을 한 번만 적어둔다
정보를 다 넣었는데 답이 더 나빠짐 컨텍스트 오염 — 필요한 것만 넣고 무관한 정보·옛 대화는 뺀다
AI가 "그 기능 없다"고 우김 학습 시점까지만 알기 때문. "없다 말고 공식 문서를 조사해 제대로 하라"고 먼저 시키고 CLAUDE.md에 명시
